RegReport — Documentation

Tableau de bord de regulatory reporting multi-entités : suivi, analyse et comparaison des KPI réglementaires (CET1, LCR, NSFR, Leverage Ratio), des échéances, des reportings quotidiens et des projets.

React 18 + TypeScript Vite Tailwind CSS Recharts .NET 8 + EF Core SQL Server

1Vue d'ensemble

TypeSingle Page Application (SPA) React, 100 % front-end
StackReact 18 + TypeScript, Vite, Tailwind CSS, React Router (hash), Recharts
DonnéesAujourd'hui : localStorage du navigateur. Demain : API REST (voir partie back-end)
ExportPDF des rapports KPI (jsPDF + html2canvas, chargés à la demande)

L'application est modulaire : chaque page correspond à un module métier indépendant, accessible depuis la barre de navigation.

4Le flux de données (important)

C'est le cœur de l'application. Tout passe par un état central unique.

constants.ts (seed)
      │
      ▼
dataRepository.load() ──► DataContext (state: CentralData) ──► useData() ──► Pages
      ▲                            │
      │                            ▼ (modification via setData)
dataRepository.save() ◄──── persistance auto (debounce 400 ms)
  1. Au démarrage, DataProvider appelle dataRepository.load(). Le repository choisit sa source selon VITE_API_BASE_URL : vide → mode local (lecture localStorage ou seed) ; renseignée → mode API (GET {base}/data).
  2. Les données sont stockées dans un seul objet CentralData exposé via le hook useData().
  3. Chaque page lit data et le modifie via setData(...).
  4. À chaque modification, un effet persiste automatiquement (debounce 400 ms) : localStorage.setItem en local, ou PUT {base}/data en mode API.
À retenirAucun composant ne connaît la source des données. Pour passer au back-end, on ne change que la variable VITE_API_BASE_URL.

Le hook useData()

const {
  data,            // CentralData : toutes les données de l'app
  setData,         // pour modifier les données
  allEntities,     // string[] : entités présentes (dérivé)
  allDates,        // string[] : dates présentes (dérivé, triées desc.)
  getKpisForDate,  // (entity, date, currency?) => CalculatedKpis | null
  isLoading,       // chargement initial en cours
  loadError,       // erreur de chargement (mode API)
  mode,            // 'local' | 'api'
} = useData();

KPI : données brutes vs données calculées

  • KpiHistoryEntry = ce qui est saisi/importé (capital CET1, RWA crédit / marché / opérationnel, exposition, liquidité HQLA/NCO par devise…).
  • CalculatedKpis = ce qui est dérivé par calculateKpis() :
cet1cet1Capital / rwaTotal × 100
lcrhqla / netCashOutflows × 100
nsfrasf / rsf × 100
leveragetier1 / exposure × 100

C'est getKpisForDate(entity, date, currency) qui fait le pont entre les deux.

6.2 KPI Analysis /details — le module le plus riche

  • Sélection entité / date / devise / période et date de comparaison.
  • Onglets Overview / Capital / Liquidity.
  • Cartes KPI avec seuils risk-appetite (red/amber/green) et graphes d'évolution.
  • Waterfalls : évolution du ratio CET1, du RWA, du LCR entre deux dates.
  • Tables de composition (capital CET1, RWA), top contreparties.
  • Export PDF du rapport affiché (jsPDF / html2canvas chargés dynamiquement à ce moment-là).

6.4 Deadlines /deadlines

Liste triable et calendrier visuel ; chaque échéance a un statut (completed / inprogress / upcoming), un historique de changements et des pièces jointes. Calcul de l'échéance réglementaire (+10 jours ouvrés).

6.6 Data Management /datamanagement

  • Import CSV/JSON pour KPI, liquidité, contreparties, grands risques… (modes replace ou append/merge).
  • Export des données et templates CSV.
  • Diagnostic via diagnosisService.ts (variations anormales, densité RWA élevée, valeurs négatives suspectes…).
  • Réinitialisation des données (localStorage.removeItem).

7Le système de design (style « EFG »)

La charte visuelle s'inspire d'une présentation corporate de banque privée : sobre, aérée, titres en graisse légère, une seule couleur d'accent (rouge profond) utilisée avec parcimonie.

  • Palette : définie une seule fois dans src/theme.ts (PALETTE, CHART_COLORS, STATUS_COLORS) et reflétée par nom dans tailwind.config.js. Pour changer toute l'identité visuelle, modifier ces deux fichiers.
  • Composants de mise en page : PageHeader, SectionHeader (titre + suffixe muet), BulletList (puces carrées rouges), Card, Modal, InfoBox, Select, TabButton.
  • Charts (Recharts) : toutes les couleurs viennent de CHART_COLORS / PALETTE. Grilles en filet clair, hauteurs compactes, rouge réservé à l'emphase / aux totaux.

8Persistance & réinitialisation

  • Mode local : les données vivent dans localStorage sous la clé regReportData. Vider le cache (ou « Reset » dans Admin) revient au seed de constants.ts.
  • Mode API : voir la partie back-end ci-dessous.

9Conventions & bonnes pratiques

  • TypeScript strict : npm run lint doit rester vert.
  • Lazy-loading : chaque page est un chunk séparé. Les libs lourdes (jsPDF, html2canvas) sont importées dynamiquement.
  • Couleurs : ne jamais coder un hex en dur — utiliser theme.ts (charts) ou les tokens Tailwind brand-* (classNames).
  • Données : toujours passer par useData() / setData(), jamais lire localStorage directement depuis une page.

Connecter un back-end

.NET 8 + Entity Framework Core + SQL Server

Comment remplacer le stockage localStorage par un vrai back-end ASP.NET Core Web API, et y brancher le front existant sans réécrire les composants. Le front est déjà prêt : dataRepository.ts bascule en mode API dès que VITE_API_BASE_URL est définie.

AArchitecture cible

┌──────────────────┐   HTTPS / JSON   ┌────────────────────────┐      ┌────────────┐
│ Front React/Vite │ ───────────────► │  ASP.NET Core Web API  │ ───► │ SQL Server │
│ dataRepository   │ ◄─────────────── │  Controllers + EF Core │ ◄─── │  (MSSQL)   │
└──────────────────┘                  └────────────────────────┘      └────────────┘
     mode 'api'                          /api/data, /api/kpis, ...

Deux stratégies de stockage, au choix :

  • Option A — « Document » (démarrage rapide) : une seule table qui stocke le JSON CentralData complet. Fonctionne immédiatement avec GET/PUT /api/data.
  • Option B — Relationnel (cible) : un schéma SQL normalisé (une table par ressource) pour requêter, auditer et sécuriser finement.
AstuceOn peut commencer par A puis migrer vers B sans toucher au front : le front parle toujours /api/data, c'est le contrôleur qui change d'implémentation.

BLe contrat que le front attend déjà

MéthodeEndpointCorpsRéponse
GET{base}/data—CentralData (JSON)
PUT{base}/dataCentralData204 No Content
SuffisantImplémenter ces deux endpoints suffit pour que toute l'application fonctionne sur le back-end. Les endpoints granulaires sont une amélioration ultérieure.

Option A — Contrôleur /api/data (document)

[ApiController]
[Route("api/data")]
public class DataController : ControllerBase
{
    private readonly AppDbContext _db;
    public DataController(AppDbContext db) => _db = db;

    [HttpGet]
    public async Task<IActionResult> Get()
    {
        var doc = await _db.CentralDataDocuments.FindAsync(1);
        return Content(doc?.Payload ?? "{}", "application/json");
    }

    [HttpPut]
    public async Task<IActionResult> Put()
    {
        using var reader = new StreamReader(Request.Body);
        var json = await reader.ReadToEndAsync();
        var doc = await _db.CentralDataDocuments.FindAsync(1)
                  ?? _db.CentralDataDocuments.Add(new() { Id = 1 }).Entity;
        doc.Payload = json;
        doc.UpdatedAtUtc = DateTime.UtcNow;
        await _db.SaveChangesAsync();
        return NoContent();
    }
}

DOption B — Schéma relationnel (cible)

Schéma normalisé miroir de types.ts. Les structures imbriquées (liquidité par devise, breakdowns, historique/pièces jointes) sont stockées en colonnes JSON (NVARCHAR(MAX)) — SQL Server sait les requêter via JSON_VALUE / OPENJSON.

-- KPI bruts (un enregistrement par entité + date)
CREATE TABLE dbo.KpiHistory (
    Id           INT IDENTITY PRIMARY KEY,
    Entity       NVARCHAR(64)  NOT NULL,
    [Date]       DATE          NOT NULL,
    Cet1Capital  DECIMAL(18,2) NOT NULL,
    CreditRWA    DECIMAL(18,2) NOT NULL,
    MarketRWA    DECIMAL(18,2) NOT NULL,
    OpRWA        DECIMAL(18,2) NOT NULL,
    Tier1        DECIMAL(18,2) NOT NULL,
    Exposure     DECIMAL(18,2) NOT NULL,
    Cet1CapitalBreakdown NVARCHAR(MAX) NULL,   -- JSON
    Liquidity            NVARCHAR(MAX) NULL,   -- JSON (par devise)
    CONSTRAINT UQ_KpiHistory UNIQUE (Entity, [Date])
);

-- + Deadlines, CounterpartyRwa, LargeExposures, TeamMembers,
--   Projects, ProjectTasks, AppSettings (clé/valeur JSON)

Mappez chaque table à une entité EF Core et exposez un DbSet<> par ressource. Le contrôleur /api/data devient alors un agrégateur qui recompose CentralData à partir des tables (et inversement au PUT).

FSécurité (recommandé en banque)

  • Authentification : Microsoft Entra ID (Azure AD) via OpenID Connect. API : AddAuthentication().AddJwtBearer(...) + [Authorize]. Front : access token (MSAL.js) injecté dans ApiRepository.headers() (emplacement déjà préparé).
  • HTTPS obligatoire (app.UseHttpsRedirection()).
  • Audit : les colonnes UpdatedBy / UpdatedAtUtc tracent les écritures.
  • Validation du payload côté serveur (DataAnnotations / FluentValidation) avant persistance.

GPointer le front sur l'API — pas à pas

  1. Lancer l'API .NET (dotnet run) et noter son URL (ex. https://localhost:7042).
  2. À la racine du front, créer .env.local (copie de .env.example) : 
    VITE_API_BASE_URL=https://localhost:7042/api
  3. Redémarrer le serveur de dev : npm run dev.
  4. Le front passe automatiquement en mode api : GET /api/data au démarrage, PUT /api/data à chaque modification (debounce 400 ms).
  5. Vérifier dans l'onglet Network que les appels partent vers l'API. En cas d'échec, un bandeau rouge s'affiche et l'app retombe sur le seed local.
Revenir en localVider VITE_API_BASE_URL (ou supprimer .env.local) et redémarrer.
